## PRPilot — build context + operating prompt (Cursor)

You are building **PRPilot**, an AI-powered PR review product (GitHub App first) that:
- listens to PR webhooks
- queues async review jobs (do not block webhook)
- fetches diffs/files from GitHub
- calls OpenAI with a strict prompt stack
- posts inline review comments + summary
- posts a GitHub Status Check (“merge readiness score”)
- persists findings for analytics + weekly manager digest
- supports India-first billing (INR + GST) as a differentiator

The HTML files in this folder are the **product/architecture roadmap**:
- `prpilot_architecture.html`: system architecture + stack + core code sketch + sprint plan + unit economics
- `prpilot_prerequisites.html`: setup checklist for accounts/tools + local dev prerequisites
- `prpilot_prompt_vault.html`: production prompt stack + tiers (core, manager, junior coach, security audit, score)

This `.cursor.md` is the **single source of truth** for how to implement PRPilot without re-prompting: stack, repo structure, conventions, and non-negotiable engineering constraints.

### North-star product decisions (treat as requirements)

- **Integration**: GitHub App (not OAuth) as primary. GitHub Actions optional add-on later.
- **Latency**: webhook handler must respond fast (\(<10s\)). All AI work runs async in workers.
- **Noise policy**: suppress LOW-severity findings by default; prefer fewer, higher-signal comments.
- **Output contract**: core review returns **valid JSON** matching schema; UI/worker code must treat output as untrusted and validate/guard.
- **Pricing moat**: India-first billing (INR + GST invoices) and manager digest.

### Target stack (MVP default)

- **Backend**: Node.js 20+ + TypeScript + Fastify
- **GitHub integration**: `@octokit/app` (GitHub App auth + webhooks)
- **Queue**: Redis + BullMQ
- **DB**: PostgreSQL (Supabase recommended)
- **Frontend**: Next.js (App Router) dashboard, deployed on Vercel
- **Email**: Resend (digest + transactional)
- **Payments**: Razorpay (IN) + Stripe (international)
- **AI**:
  - core review: `gpt-4o-mini` (cheap/fast)
  - security audit + manager digest: `gpt-4o` (quality)

### Repository layout (create/maintain this structure)

Monorepo, two apps + shared library:
- `apps/backend/`
  - `src/server.ts` (Fastify bootstrap)
  - `src/github/` (webhook handler, app auth, GitHub API wrapper)
  - `src/queue/` (BullMQ queue + job definitions)
  - `src/workers/` (review worker, digest worker)
  - `src/ai/` (OpenAI client, prompt routing, schema validation)
  - `src/db/` (Supabase client / SQL access, repositories)
  - `src/config/` (env parsing + validation)
  - `src/lib/` (shared utils: logging, retry, rate limits)
- `apps/web/`
  - Next.js dashboard, auth, settings, billing, reports
- `packages/shared/`
  - Types (DTOs), JSON schemas, common constants

If repo is still empty, start by scaffolding this structure and commit early.

### Backend contracts (do not violate)

- **Webhook endpoint**: verify GitHub signature (HMAC) and immediately enqueue job.
- **Job payload** must include: `{ owner, repo, prNumber, installationId, headSha, prTitle, prBody, author, files[] }`
- **Diff fetching**: use GitHub API (`pulls.listFiles`) and store patches; handle missing `patch` (binary/too large).
- **OpenAI output**: must be parsed as JSON; handle parse failures with safe fallback comment (ask to retry / post summary only).
- **Comment posting**:
  - single `createReview` with summary + inline comments (filter by severity threshold)
  - do not spam; cap comments (e.g. top N by severity) when large PRs
- **Status check**: always post a status with score + recommendation. CI failing overrides score.
- **Rate limiting**: per org and per repo to avoid cost overruns; queue should backpressure.

### Prompt stack (source of truth = Prompt Vault page)

Implement prompts as versioned templates (store in `packages/shared/prompts/`):
- `system/core-system.md` (persona + taxonomy + golden rules)
- `review/core-json.md` (core review JSON schema)
- `digest/manager-weekly.md` (Markdown output)
- `coach/junior.md` (coaching wrapper)
- `security/owasp-json.md` (OWASP audit JSON)
- `score/merge-readiness-json.md` (status check JSON)

Rules:
- Core review must return **ONLY JSON** (no prose)
- Security audit returns **ONLY JSON**
- Digest returns Markdown (for email/Slack)

### Database model (minimum)

Create tables (or equivalent):
- `organizations` (github installation, plan, settings)
- `repositories` (org mapping)
- `pull_requests` (repo, number, headSha, metadata)
- `findings` (label, severity, file, line_range, title, explanation, fix, etc.)
- `users` (auth + org membership)
- `subscriptions` (plan, provider, status)

### Environment variables (backend)

Never commit secrets. Use `.env` locally and platform env vars in prod.

Required:
- `GITHUB_APP_ID`
- `GITHUB_PRIVATE_KEY` (multiline; support `\n` escaped)
- `GITHUB_WEBHOOK_SECRET`
- `OPENAI_API_KEY`
- `REDIS_URL` (or host/port)
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY` (backend only)
- `DATABASE_URL` (if using direct Postgres)

Optional (later):
- `RESEND_API_KEY`
- `RAZORPAY_KEY_ID` / `RAZORPAY_KEY_SECRET`
- `STRIPE_SECRET_KEY`
- `APP_BASE_URL` (for links in comments/emails)

### Engineering standards (how you should implement)

- Prefer **typed boundaries**: validate webhook payloads, job payloads, and AI JSON with schemas.
- Be conservative about **cost**: short prompts, limit diff size, suppress LOW, cap comment count.
- Handle **large PRs** explicitly: if \(\>500\) lines changed, post “too large to review safely” + top critical findings only.
- Logging must be structured; never log secrets or full diffs in prod.
- No framework churn: stick to stack above unless explicitly asked to change it.

### Execution roadmap (what to build first)

MVP “Day 1 pipe” definition:
- GitHub App receives `pull_request.opened` and `pull_request.synchronize`
- queues job
- worker fetches diff
- calls OpenAI core review
- posts summary + limited inline comments
- posts status check score

Then:
- persist findings to DB
- dashboard to view PR history + findings
- weekly manager digest job (cron) + email/slack delivery
- security audit trigger for sensitive paths
- billing + plan gates

### When to ask for clarification (only when necessary)

Ask only if a change requires choosing between materially different product paths, e.g.:
- mono-repo vs split repos
- Supabase-only vs direct Postgres migrations
- deployment target (Railway vs VPS vs AWS)
- pricing/plan gating specifics
